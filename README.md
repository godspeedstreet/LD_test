# Часть 1. Поиск блогеров для бартера
## AI-инструмент поиска блогеров

Инструмент закрывает полный цикл из тестового задания: анализирует базу блогеров,
которые компания уже считает эталонными → строит из них «портрет идеального
блогера» → ищет новых похожих кандидатов → пишет каждому персональный оффер
о бартерном сотрудничестве. 

ДЭМО ВЕРСИЯ - в этом проекте показано как программа будет реально выполнять свой алгоритм со всеми существующими и необходимыми данными. 

## Запуск 

### 1. Установка

```bash
скачать репозиторий любым способом
открыть терминал от имени администратора
ввести команду 'pip install -r requirements.txt'
затем 'python main.py --step all --limit 5'
```

### 2. Что нужно вписать в `.env`

| Переменная | Зачем нужна | Где взять |
|---|---|---|
| `SPREADSHEET_ID` | ID таблицы блогеров — часть ссылки между `/d/` и `/edit` | Уже есть значение по умолчанию в `.env.example`, при необходимости заменить на свою таблицу |
| `OPENAI_API_KEY` | Анализ профилей, ранжирование кандидатов, генерация офферов | platform.openai.com → API keys |
| `OPENAI_MODEL` | Модель для всех LLM-вызовов | По умолчанию `gpt-4o-mini` — этого достаточно |



### 3. Результат

`output/latest_run.json` — портрет, список новых кандидатов и готовые тексты
офферов по каждому. Пример структуры:

```json
{
  "portrait": { "visual_style": "...", "search_keywords": ["..."] , "..." : "..." },
  "source_profiles_count": 5,
  "new_candidates": [ { "username": "...", "why_fit": "..." } ],
  "offers": [ { "username": "...", "message": "..." } ]
}
```

## Как устроен пайплайн

```
sheet_reader.py        — читает таблицу, достаёт username'ы из ссылок/заголовков
instagram_collector.py — собирает био/подписчиков/подписи по существующим блогерам
portrait_builder.py    — LLM строит «портрет идеального блогера» (JSON)
blogger_discovery.py   — ищет новых кандидатов по хэштегам из портрета + LLM ранжирует
offer_generator.py     — LLM пишет персональный оффер под каждого кандидата
main.py                — оркестрирует всё, кэширует промежуточные результаты
cache.py                — сохранение/загрузка между шагами и fixture-данные для демо
```

Промпты лежат отдельными файлами и подгружаются в рантайме, а не зашиты в код:

- `01_portrait_prompt.md` — анализ базы, построение портрета
- `02_discovery_prompt.md` — правила поиска и ранжирования новых кандидатов
- `03_offer_prompt.md` — генерация персонального оффера

# Часть 2. Автоматизация генерации описаний карточек товара

Я бы взял генерацию текстовых описаний карточек товара для Wildberries и Ozon. При команде 60+ человек и полном цикле от эскиза до готового товара это узкое место почти всегда одно и то же: на каждую новую позицию (а в женской одежде это часто десятки SKU в неделю — размерные сетки, цветовые варианты) нужен текст, который одновременно закрывает SEO-требования площадки, читается живо, а не как машинный перевод характеристик, и не противоречит реальным параметрам товара. Обычно это делает контент-менеджер вручную по шаблону, тратя 15–20 минут на карточку, и с ростом ассортимента это первое, что начинает тормозить весь конвейер публикаций — не пошив и не логистика, а текст.

Автоматизировать я бы стал не «нажал кнопку — получил готовое описание», а связку человек+ИИ с чётким разделением ролей: ИИ генерирует первый черновик и SEO-ключи, менеджер проверяет фактуру и публикует. Полная автономность здесь опасна: если модель один раз напишет «состав: 100% хлопок» для вещи с эластаном, это репутационный и юридический риск, а не мелкая правка. Поэтому в качестве входных данных нужна не свободная генерация «из головы», а строго структурированная карточка: состав ткани, размерная сетка, цвет, категория, ключевые особенности кроя из внутренней системы (Google Sheets/1С/PIM — что у них используется), плюс фото товара для мультимодальной модели, чтобы описание визуальных деталей (посадка, драпировка, фактура) соответствовало тому, что реально видно на фото, а не додумывалось.

Инструменты: LLM с поддержкой изображений (GPT-4o/Claude с vision) для самого текста, отдельный промпт-шаблон под каждую площадку — у Wildberries и Ozon разные ограничения по длине и разная логика ранжирования по ключевым словам в описании, поэтому один универсальный текст даёт просадку в обеих выдачах одновременно. Для SEO-части нужна интеграция с сервисом парсинга поисковых запросов маркетплейсов (MPStats, Moneyplace или аналог) — иначе модель будет расставлять ключевые слова наугад, без опоры на то, что реально ищут покупатели. Результат уходит не сразу в публикацию, а в очередь на модерацию — с подсветкой, что сгенерировано автоматически, чтобы у менеджера был быстрый чек-лист: соответствует ли состав, размеры, нет ли выдуманных деталей.

Проверял бы решение в два этапа. Сначала — на исторических данных: беру 50–100 уже опубликованных карточек, для которых есть метрики (конверсия в просмотр, конверсия в корзину), прогоняю через инструмент и сравниваю сгенерированный текст с оригиналом вслепую — даю менеджерам оценить оба варианта, не говоря, где чей, по шкале «купил бы/не купил бы» и «доверяю ли составу и размерам». Если автоматический вариант проигрывает по доверию — это сигнал дорабатывать промпт до запуска в бой, а не после. Дальше — A/B на новых карточках: часть новых SKU получает описание от ИИ (с проверкой менеджера), часть — полностью ручное, и через две-три недели сравниваю по метрикам площадки.

Метрика успеха через месяц — не «сколько текстов сгенерировано» (это тщеславная метрика, она растёт сама по себе и ничего не говорит об эффекте), а среднее время выхода новой карточки в публикацию от загрузки фото до размещения на площадке, и отдельно — конверсия карточка→корзина у SKU с ИИ-описанием против ручных за тот же период. Если время сокращается в 2–3 раза (условно с 15–20 минут до 5–7 на карточку с учётом проверки), а конверсия не проседает или растёт — решение оправдано и можно масштабировать на весь ассортимент. Если конверсия падает даже при выигрыше по времени — проблема в качестве текста, и это повод пересмотреть промпт или вернуть больше контроля человеку, а не тянуть автоматизацию любой ценой.
