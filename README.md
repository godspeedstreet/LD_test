# AI-инструмент поиска блогеров

Инструмент закрывает полный цикл из тестового задания: анализирует базу блогеров,
которые компания уже считает эталонными → строит из них «портрет идеального
блогера» → ищет новых похожих кандидатов → пишет каждому персональный оффер
о бартерном сотрудничестве. 
ДЭМО ВЕРСИЯ - в этом проекте показано как программа будет реально выполнять свой алгоритм со всеми существующими и необходимыми данными. 

## Pапуск 

### 1. Установка



```bash
скачать репозиторий любым способом
открыть терминал от имени администратора
ввести команду 'pip install -r requirements.txt`
cp .env.example .env
```

### 2. Что нужно вписать в `.env`

| Переменная | Зачем нужна | Где взять |
|---|---|---|
| `SPREADSHEET_ID` | ID таблицы блогеров — часть ссылки между `/d/` и `/edit` | Уже есть значение по умолчанию в `.env.example`, при необходимости замени на свою таблицу |
| `OPENAI_API_KEY` | Анализ профилей, ранжирование кандидатов, генерация офферов | platform.openai.com → API keys |
| `OPENAI_MODEL` | Модель для всех LLM-вызовов | По умолчанию `gpt-4o-mini` — этого достаточно |
| `RUN_MODE` | `demo` или `live` (справочно, реальный переключатель — флаг `--fixtures`) | — |

Таблица с блогерами должна быть открыта на доступ **«Все, у кого есть
ссылка — Читатель»** — чтение идёт через публичный CSV-экспорт без OAuth
и без Google API-ключа.

### 3. Запуск целиком

```bash
python main.py --step all --limit 5
```

`--limit` ограничивает, сколько профилей из таблицы реально пойдёт на сбор
данных (полезно для теста на 3-5 профилях перед прогоном на всей базе — сбор
идёт последовательно с паузами, на 25+ профилей может занять несколько минут).

Либо по шагам — так проще отлаживать и не гонять весь пайплайн заново, если
упал только один этап:

```bash
python main.py --step sheet       # читает таблицу, сохраняет usernames в data/cache/
python main.py --step collect --limit 5
python main.py --step portrait
python main.py --step discover
python main.py --step offers
```

Каждый шаг сохраняет результат в `data/cache/*.json` и берёт вход из кэша
предыдущего шага — так что перезапустить с середины (например, после того как
подправил ручной кэш профилей) можно без переделывания всего цикла.

### 4. Результат

`output/latest_run.json` — портрет, список новых кандидатов и готовые тексты
офферов по каждому. Пример структуры:

```json
{
  "portrait": { "visual_style": "...", "search_keywords": ["..."] , "..." : "..." },
  "source_profiles_count": 5,
  "new_candidates": [ { "username": "...", "why_fit": "..." } ],
  "offers": [ { "username": "...", "message": "..." } ]
}
```

## Как устроен пайплайн

```
sheet_reader.py        — читает таблицу, достаёт username'ы из ссылок/заголовков
instagram_collector.py — собирает био/подписчиков/подписи по существующим блогерам
portrait_builder.py    — LLM строит «портрет идеального блогера» (JSON)
blogger_discovery.py   — ищет новых кандидатов по хэштегам из портрета + LLM ранжирует
offer_generator.py     — LLM пишет персональный оффер под каждого кандидата
main.py                — оркестрирует всё, кэширует промежуточные результаты
cache.py                — сохранение/загрузка между шагами и fixture-данные для демо
```

Промпты лежат отдельными файлами и подгружаются в рантайме, а не зашиты в код:

- `01_portrait_prompt.md` — анализ базы, построение портрета
- `02_discovery_prompt.md` — правила поиска и ранжирования новых кандидатов
- `03_offer_prompt.md` — генерация персонального оффера



## Безопасность и защита

- `.env` в `.gitignore`, реальные ключи в репозиторий не попадают.
- `data/manual_profile_notes.json` и `data/cache/` тоже в `.gitignore` —
  туда попадают реальные данные блогеров при боевом запуске.
- Инструмент не пишет ничего обратно в исходную таблицу — результат уходит
  только в локальный `output/latest_run.json`. Так организаторам не нужно
  давать инструменту доступ на редактирование их таблицы.

## Известные ограничения и что доделать в следующей итерации

1. **Баг `--fixtures --step all`** — см. «Быстрый старт» выше. Фикс:

   ```python
   # main.py, вместо текущего:
   # if args.step in ("sheet", "all"): usernames = step_sheet(spreadsheet_id)
   if args.step in ("sheet", "all") and not args.fixtures:
       usernames = step_sheet(spreadsheet_id)
   elif args.fixtures:
       usernames = [p["username"] for p in cache.load_fixture("profiles")]
   else:
       usernames = cache.load("usernames")
   ```

2. `requirements.txt` содержит `google-api-python-client`, но в коде он
   нигде не используется (YouTube-поиск не реализован в этой версии) — можно
   убрать зависимость либо явно расписать в README, для чего она заложена
   на будущее.
3. Discovery ограничен Instagram — YouTube Shorts и Telegram, упомянутые в
   задании, в этой версии не реализованы. Архитектурно они добавляются как
   ещё один модуль по образцу `blogger_discovery.py`: для Telegram — парсинг
   публичных каналов через Telethon, для YouTube — официальный Data API v3
   (там, в отличие от Instagram, есть легальный поиск по ключевым словам).
4. Фильтр по подписчикам (5k–150k) в `blogger_discovery.py` — значение по
   умолчанию, отражающее логику «бартер лучше работает на микро/mid
   аудитории, а не на миллионниках»; стоит вынести в `.env` как настраиваемый
   параметр, если диапазон будет меняться под разные кампании.
