# Часть 1. Поиск блогеров для бартера
## AI-инструмент поиска блогеров

Инструмент закрывает полный цикл из тестового задания: анализирует базу блогеров,
которые компания уже считает эталонными → строит из них «портрет идеального
блогера» → ищет новых похожих кандидатов → пишет каждому персональный оффер
о бартерном сотрудничестве. 

>*ДЭМО ВЕРСИЯ - в этом проекте показано как программа будет реально выполнять свой алгоритм со всеми существующими и необходимыми данными. Сейчас это выглядит слишком на скорую руку, реальный проект будет отличаться по оформлению и немного по логике.*

## Запуск 

### 1. Установка

```bash
скачать репозиторий любым способом
открыть терминал от имени администратора
ввести команду 'pip install -r requirements.txt'
затем 'python main.py --step all --limit 5'
```

### 2. Как заполнить `.env`

| Переменная | Для чего | Где взять |
|---|---|---|
| `SPREADSHEET_ID` | ID таблицы блогеров — часть ссылки между `/d/` и `/edit` | Уже есть значение по умолчанию в `.env.example`, при необходимости заменить на свою таблицу |
| `OPENAI_API_KEY` | Анализ профилей, ранжирование кандидатов, генерация офферов | platform.openai.com → API keys |
| `OPENAI_MODEL` | Модель для всех LLM-вызовов | По умолчанию `gpt-4o-mini` — этого достаточно |



### 3. Результат

`output/latest_run.json` — портрет, список новых кандидатов и готовые тексты
офферов по каждому. Пример структуры:

```json
{
  "portrait": { "visual_style": "...", "search_keywords": ["..."] , "..." : "..." },
  "source_profiles_count": 5,
  "new_candidates": [ { "username": "...", "why_fit": "..." } ],
  "offers": [ { "username": "...", "message": "..." } ]
}
```

### Как устроен пайплайн

```
sheet_reader.py        — читает таблицу, достаёт username'ы из ссылок/заголовков
instagram_collector.py — собирает био/подписчиков/подписи по существующим блогерам
portrait_builder.py    — LLM строит «портрет идеального блогера» (JSON)
blogger_discovery.py   — ищет новых кандидатов по хэштегам из портрета + LLM ранжирует
offer_generator.py     — LLM пишет персональный оффер под каждого кандидата
main.py                — оркестрирует всё, кэширует промежуточные результаты
cache.py                — сохранение/загрузка между шагами и fixture-данные для демо
```

Промпты лежат отдельными файлами и подгружаются в рантайме, а не зашиты в код:

- `01_portrait_prompt.md` — анализ базы, построение портрета
- `02_discovery_prompt.md` — правила поиска и ранжирования новых кандидатов
- `03_offer_prompt.md` — генерация персонального оффера

Если возникли трудности с запуском - [видео с работой программы](https://drive.google.com/file/d/1lDuhbM-UPttNUfgrIV-6agkBBQZXYwEo/view?usp=sharing)





# Часть 2. Автоматизация генерации описаний карточек товара

Я бы выбрал автоматизацию генерации описаний товаров для Wildberries и Ozon. Мне кажется, именно на этом этапе часто теряется много времени. Пока товар уже готов, фотографии сделаны и характеристики заполнены, ещё нужно написать нормальное описание. Особенно если выходит сразу несколько размеров и цветов одной модели. На каждую карточку приходится тратить 15-20 минут, а при большом количестве новых товаров это начинает замедлять публикацию всего ассортимента.
При этом я бы не стал полностью отдавать эту задачу ИИ. На мой взгляд, лучше использовать его как помощника. Он может быстро подготовить первый вариант текста и подобрать ключевые слова, а контент-менеджер уже проверит описание перед публикацией. Это безопаснее, потому что ИИ может ошибиться в характеристиках товара. Например, неверно указать состав ткани или придумать деталь, которой на самом деле нет.

Чтобы такой инструмент работал, ему нужно передавать не только название товара, а нормальную карточку с данными: категорию, состав ткани, цвета, размеры, особенности модели и фотографии. Тогда описание будет строиться на реальной информации, а не на предположениях ИИ.
Для реализации можно использовать современные языковые модели - chatGPT или Claude. Также понадобятся отдельные шаблоны для Wildberries и Ozon, потому что требования к описаниям у площадок немного отличаются. Если компании важно SEO, то дополнительно стоит использовать сервисы аналитики маркетплейсов, например MPStats или Moneyplace, чтобы добавлять в описание реальные поисковые запросы покупателей, а не случайные ключевые слова.

Проверять эффективность такого решения я бы начал на уже опубликованных товарах. Например, взял бы несколько десятков карточек, сгенерировал для них новые описания и дал менеджерам сравнить их с существующими, не говоря, какой вариант написал человек, а какой - ИИ. Если новые тексты выглядят не хуже и не содержат ошибок, можно переходить к тестированию на новых товарах. После этого часть карточек публиковать с описаниями, подготовленными с помощью ИИ (после проверки менеджером), а часть - полностью вручную, чтобы сравнить результаты.
Через месяц я бы смотрел не на количество сгенерированных текстов, а на более полезные показатели. В первую очередь — насколько сократилось время подготовки карточки к публикации. Если раньше на описание уходило около 15–20 минут, а теперь вместе с проверкой требуется 5–7 минут. Второй показатель - конверсия карточки в добавление товара в корзину. Если она осталась на прежнем уровне или стала выше, значит автоматизация действительно помогает компании, а не просто экономит время. Если же конверсия заметно снизится, значит качество описаний нужно дорабатывать и усиливать проверку со стороны человека.


## Часть 3. Прошлые работы

Можно посмотреть на мои закрепленные работы в [профиле](https://github.com/godspeedstreet). Также до недавнего времени существовало несколько Telegram-ботов, но из-за завершения сотрудничества с заказчиком я прекратил их поддержку:

- Бот по записи на занятия спортом - пользователь выбирает время и название тренировки, бот спрашивает контактную информацию и передает админам, каждому админу приходит уведовмление о новой записи.
- Также бот для сбора и переработки новосных постов - автоматический сбор новостных публикаций через юзербота, переписывание текста с помощью ИИ и публикация в выбранные Telegram-каналы по нажатию одной кнопки админа.

Все проекты выполнял один с нуля при использовании нейросети.
